# xfchat_cli Agent 办公场景使用指南

本文面向已经接入 `xfchat_cli` 的 Agent。目标不是解释底层命令，而是说明：

- 现在已经能组合出哪些典型办公场景
- 应该怎样向 Agent 描述需求
- 哪些场景适合直接用真实数据执行
- 哪些能力当前还有边界

这里的 `xfchat_cli` 即当前私有化 lark-cli fork；二进制名和配置目录统一为 `xfchat_cli` / `~/.xfchat_cli`。邮件能力当前不在运行态矩阵内。

## 1. 先说结论

当前这套私有化飞书能力，已经足够让 Agent 在真实办公环境里完成这几类任务：

- 消息协同：发消息、回消息、留痕、群内通知、私聊通知
- 文档产出：生成正式文档、追加内容、覆盖内容、插入图片/文件、写结构化块
- 数据整理：建多维表格、建电子表格、写记录、更新数据、生成演示样板
- 协作管理：日程创建、任务创建、任务分派、表单与协作者管理
- 组织查询：查人、查知识库节点、根据对象链接反查底层 token

更适合当前阶段的用法是：

- Agent 负责理解你的自然语言目标
- 自动拆成 `Docs / Sheets / Base / IM / Calendar / Task / Contact` 的组合操作
- 最后把结果落到飞书里，并在群里留痕

## 2. 使用原则

### 2.1 适合 Agent 直接执行的任务

- “帮我写一篇周报，并发到群里”
- “把这个项目排期整理成多维表格，再生成一份汇报文档”
- “把今天和明天的日程、未完成任务汇总成日报”
- “找一下某个同事的 open_id，再把任务分给他”
- “把群里留痕的测试结果整理成一份可演示文档”

### 2.2 不适合一句话直接丢给 Agent 的任务

- “把所有东西都自动做好”
- “帮我把知识库搭起来”
- “帮我同步全部飞书数据”

这些目标太大，最好改成：

- 指定目标对象
- 指定结果形式
- 指定落盘位置
- 指定是否需要群内留痕

## 3. 推荐描述方式

给 Agent 下任务时，尽量说清 4 件事：

1. 目标
2. 数据来源
3. 输出形式
4. 是否留痕

推荐模板：

```text
帮我基于 <数据来源>，完成 <目标>。
结果输出到 <Docs/Base/Sheets/Task/Calendar/IM>。
如果适合，请顺手在 <群聊/私聊> 留痕。
优先用真实数据，不要造假数据。
```

更具体的模板：

```text
帮我读取真实数据，整理成一份给老板看的汇报。
需要：
1. 一份云文档正文
2. 一份电子表格或多维表格作为附件或引用
3. 最后发到指定群里留痕
```

## 4. 典型办公场景

### 场景 A：项目周报

Agent 可以做的事：

- 从多维表格读取项目排期样板
- 从任务模块读取当前任务状态
- 从日历模块读取近期会议/日程
- 生成一篇结构化云文档
- 在群里发一条摘要消息并附文档链接

适合这样描述：

```text
帮我基于当前真实的项目排期、任务和日程，生成一篇项目周报。
要求：
1. 用云文档输出正式版
2. 内容要分“进展、风险、下周计划”
3. 最后发到项目群里留痕
```

建议输出组合：

- `Base`：承载项目排期
- `Task`：承载执行状态
- `Calendar`：承载会议和计划
- `Docs`：承载正式周报
- `IM`：承载通知和留痕

### 场景 B：老板演示版项目看板

Agent 可以做的事：

- 创建或更新多维表格业务样板
- 建立不同视图
- 生成对应讲解文档
- 发群消息附上演示链接

适合这样描述：

```text
帮我做一个给老板演示的项目样板。
需要：
1. 一个多维表格，字段尽量完整
2. 一篇说明文档，解释怎么读这个表
3. 最后把链接发到群里
```

真实样例：

- [星火派 Agent 软件开发排期](https://yf2ljykclb.xfchat.iflytek.com/base/basrz4CISgYO2yc3n4kfuESDLWb)
- [字段能力样板 Base](https://yf2ljykclb.xfchat.iflytek.com/base/basrzM5h3cX7nUJNz0NTX2uK1pc)

### 场景 C：会议后自动整理

Agent 可以做的事：

- 读取今天或指定时间段的日程
- 拉取相关任务
- 生成会议后待办文档
- 给参与人发群消息或私聊提醒

适合这样描述：

```text
帮我把今天的会议相关安排整理成一个行动清单。
要求：
1. 读取真实日程
2. 生成一篇文档
3. 如果有明确负责人，就在文档里 @用户
4. 最后把摘要发到群里
```

### 场景 D：数据收集 + 表单

Agent 可以做的事：

- 在 Base 里建立结构化数据表
- 建表单
- 更新协作者和权限
- 输出说明文档

适合这样描述：

```text
帮我建立一个数据收集方案。
需要：
1. 一个多维表格
2. 一个表单入口
3. 一篇说明文档
4. 最后把表单链接发给群里
```

### 场景 E：群内正式通知

Agent 可以做的事：

- 生成云文档
- 在文档里写结构化块
- 在群里发送摘要 + 链接
- 增加 Pin / Reaction 留痕

适合这样描述：

```text
帮我把这件事整理成正式通知。
要求：
1. 先生成云文档
2. 再在群里发一条精简版摘要
3. 文档链接一并带上
4. 尽量保留留痕
```

## 5. 怎么让 Agent 更稳定地工作

### 5.1 明确输出位置

不要只说“帮我整理一下”，更好的是：

- “整理成云文档”
- “整理成多维表格”
- “整理成电子表格”
- “最后发到群里”

### 5.2 明确是否允许用真实数据

如果你希望它直接使用当前账号权限读取真实信息，建议明确说：

```text
可以直接使用我当前 token 能访问到的真实数据，不要造测试数据。
```

### 5.3 明确是否需要保留痕迹

如果你是要验证链路或做演示，建议明确说：

```text
请保留留痕，不要清理消息、文档、表格或任务。
```

如果你只是想验证能力，可以说：

```text
测试后可以清理临时对象，只保留最终结果。
```

## 6. 推荐给 Agent 的指令模板

### 模板 1：文档 + 群通知

```text
帮我基于真实数据整理一份正式汇报。
输出到云文档，最后把摘要和链接发到群里。
请保留留痕。
```

### 模板 2：表格 + 文档 + 留痕

```text
帮我做一个完整的业务样板。
需要一个多维表格、一篇说明文档，最后把链接发到群里。
优先用真实数据，不够的地方再补演示样例。
```

### 模板 3：任务协作

```text
帮我把这项工作拆成任务并分配给相关人。
如果需要，先查通讯录获取正确的 open_id。
最后生成一份简短说明发给我。
```

### 模板 4：会议跟进

```text
帮我基于今天的真实日程，整理一份会议跟进清单。
输出为云文档，并给相关群发一条摘要消息。
```

### 模板 5：老板演示材料

```text
帮我做一个老板演示版样板。
需要：
1. 一份多维表格或电子表格
2. 一份讲解文档
3. 文档里带真实链接
4. 最后发群里留痕
```

## 7. 当前最适合组合的模块

### 组合 1：`Base + Docs + IM`

适合：

- 项目排期
- 业务台账
- 老板演示

### 组合 2：`Sheets + Docs + IM`

适合：

- 数据整理
- 表格汇报
- 运营周报

### 组合 3：`Calendar + Task + Docs`

适合：

- 每日站会
- 会议跟进
- 待办整理

### 组合 4：`Contact + Task + IM`

适合：

- 找人
- 分任务
- 发提醒

## 8. 当前边界

以下事项当前不要对 Agent 抱过高预期：

- Wiki 知识空间创建
- Docs 复杂块读取的完整保真
- Docs 评论创建
- IM 通过 `--user-id` 直接读取私聊历史
- Sheets 图表和 `filter_view` 全覆盖
- Base `record-history-list` 和 `data-query`

这些不是完全不能做，而是目前还不适合当“稳定办公主链路”。

## 10. 建议的下一步

如果你准备让团队开始真正使用 Agent + CLI，建议按这个顺序推广：

1. 先推广 `Docs + IM`，因为最容易形成直观结果
2. 再推广 `Base + Docs + IM`，做业务样板和演示
3. 再推广 `Calendar + Task`，做个人与团队协作
4. 最后再补 `Wiki / Drive / Sheets 图表` 这类外围能力
