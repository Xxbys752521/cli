- 不只是“给一个链接做摘要”，而是要能围绕云文档形成稳定的读取与查询能力。

### 2. 文档落盘和归档能力要补齐

相关消息：

### 3. 要把 GitHub / 代码理解能力转成办公产物

相关消息：

- `2026-01-30 10:50`：`根据我们做的东西，每天自动帮我们生成小红书风格的推文，保存在云空间下面...`

对应问题：

- 系统不能停留在“读代码”和“答问题”。
- 要能从 PR、代码变更、文档中抽取信息，转成面向业务、面向传播、面向协作的产物。
- 这类能力对内部办公提效也有价值，因为很多内部周报、纪要、方案摘要、本质上也是同类加工链路。

### 4. 现在的东西还没有形成明确场景价值

相关消息：

- `2026-03-10 23:54`：`装好了不知道干啥`
- `2026-03-10 23:54`：`目前就是没啥说法`
- `2026-03-11 10:23`：`openclaw只是一个架子`
- `2026-03-11 10:23`：`看你接哪个模型`

对应问题：

- 现在更像一个通用框架或能力架子，但对一线使用者来说，价值没有被收敛成清楚的使用场景。
- 如果不把原子能力沉淀成可直接调度的动作，再往上接什么模型，用户体感也不会稳定。

### 5. 输出质量要可控，不能只是“生成”

相关消息：

- `2026-03-11 15:15`：`真要给人看的东西，还是要自己写`

对应问题：

- 不是所有内容都适合一键生成直接给人看。
- 面向外部的内容需要更强的可控性、证据链和人工兜底。
- 面向内部办公则可以把“生成初稿 + 人工确认”作为主路径。

### 6. 最后要能沉淀成团队能力

相关消息：

- `2026-03-11 16:52`：`做完了给大伙培训一下，openclaw实现原理与实战`

- 功能做完不够，还要能讲清楚原理、边界和最佳用法。

## 五、可借鉴的实用功能清单

如果结合公司内部办公提效，而不是只盯着“招聘面试”单一场景，我建议重点吸收下面几类能力。

### 1. 记录自动结构化

- 自动转录
- 自动摘要
- 自动按模板拆章节
- 自动抽关键结论、风险、待办

适用价值：

- 面试纪要
- 需求沟通纪要
- 技术评审纪要
- 方案讨论纪要

### 2. 基于记录的二次问答

- 对历史对话、文档、纪要直接提问
- 快速找某个人提过什么问题、某个候选人说过什么、某个方案结论是什么

适用价值：

- 降低翻聊天记录和翻纪要的时间成本
- 让历史信息真正可复用

### 3. 结构化评价与证据绑定

- 结论不是一句空话
- 每个评价点尽量能回溯到原文、原对话、原片段

适用价值：

- 内部评审
- 候选人评价
- 项目复盘
- 方案优先级决策

### 4. 产物自动落盘

- 纪要、待办、周报、方案初稿自动创建到飞书
- 支持指定目录、指定父文档、指定标题层级

适用价值：

- 减少“说完了但没沉淀”的问题
- 让办公产物进入统一知识空间

### 5. 实时辅助而不是只做会后总结

- 会中提示漏问点
- 会中提示待确认项
- 会中提示风险点

适用价值：

- 面试
- 需求沟通
- 项目对齐
- 客户会议

### 6. 模板化和场景化共存

- 底层能力原子化
- 上层输出模板按场景定制

适用价值：

- 不同团队可以复用同一套底层动作
- 同时保留不同业务场景需要的输出风格

## 六、功能迭代建议

### Phase 1：先把“能用”做出来

目标：

- 让系统在内部办公场景里先稳定可用。

建议优先做：

1. 群消息和文档的统一读取能力
2. 从消息/文档中抽摘要、待办、关键问题
3. 把结果自动写回飞书文档
4. 从单次执行变成“可复用的原子动作”
5. 修正登录与账号默认态问题，避免新用户首轮体验直接失败

验收标准：

- 能从群里定位某个人提过的问题
- 能抽成待办
- 能自动生成一份结构化文档

### Phase 2：把“可追问、可回溯”补齐

目标：

- 不只是做摘要，而是让记录可以被继续使用。

建议优先做：

1. 基于群消息/文档的问答检索
2. 为结论绑定消息证据或文档证据
3. 支持“某人提过什么问题”“某项需求是什么时候提出的”这类查询
4. 补齐模型状态、会话状态、日志追踪等排障入口

验收标准：

- 不需要人工翻群记录
- 可以快速回到原始上下文

### Phase 3：把“动作链”串起来

目标：

- 从单点工具升级成办公助手。

建议优先做：

1. 消息 -> 待办
2. 消息 -> 文档
3. 文档 -> 纪要
4. 文档/消息 -> 会议日程
5. PR/代码 -> 说明文档/周报/宣传初稿
6. 面试链路里的打断、追问、上下文衔接统一纳入一条交互主链治理

验收标准：

- 一个自然语言目标可以触发多步动作
- 中间缺权限时可以补授权后继续

### Phase 4：把“质量控制”加上

目标：

- 让产物可直接进入真实工作流，而不是只能看 Demo。

建议优先做：

1. 模板体系
2. 输出风格控制
3. 人工确认点
4. 对外内容的敏感信息过滤
5. 证据引用与回链
6. 图示化表达和内部演练材料

验收标准：

- 内部文档可直接复用
- 对外内容具备人工审核接口

## 九、建议对齐议程

如果明天和凯哥对这个计划，建议议程控制在 30 分钟，按这个结构走：

1. 先确认凯哥之前提的问题里，哪些最值得优先做成主路径。
2. 确认内部办公提效要先打哪条闭环：消息到待办，还是消息到文档，还是 PR 到文档。
3. 确认哪些功能必须带证据回链，哪些可以先只做摘要。
4. 确认这套能力是继续做场景化封装，还是直接按原子动作往下沉。
