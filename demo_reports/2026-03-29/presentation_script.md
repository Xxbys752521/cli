# 10 分钟汇报发言稿

日期：2026-03-29

## 开场

今天这次汇报，我主要想把一件事情讲清楚。

就是我们后续飞书能力这一层，到底应该继续沿着现在的场景化方式往前做，还是应该切到一套统一的原子能力体系上。

我这次专门做了三部分工作。

1. 把 `lark-cli` 按官方指引在真实飞书环境里完整跑了一遍，而且不是只跑成功链路，而是按模块、按身份，把成功、失败、输入、输出都落成了 Markdown 留痕。
2. 把我们现有的 `spark` 代码形态过了一遍，重点看它现在飞书能力是怎么组织的，Agent 是怎么接进去的。
3. 结合我在 `/Users/wangqizhao/Developer/iflytek/cli` 里已经做过的一轮私有化适配，判断这条路是不是只是概念成立，还是已经到了可以继续投入的阶段。

## 结论

我先把结论放前面。

我的判断是，后续应该推动完整迁移。

不是说把 `spark` 这个业务系统简单替换掉，而是说飞书能力这一层，应该完整迁移到统一能力层上，不要再继续走“上层还是场景化封装，底层再慢慢补原子能力”这种双轨模式。

原因很简单。

现在的场景化体系和后续要建设的原子化体系，本质上是冲突的。

如果继续并存，短期看像是折中，长期看一定会形成两套能力模型、两套接入方式、两套权限处理方式，最后复杂度只会越来越高。

## 为什么会得出这个判断

现在我们的主要思路，还是先替用户想场景。

比如我们会先想，用户是不是要发消息，是不是要做群摘要，是不是要查会议，是不是要写文档，然后围绕这些场景去封装 service、封装 tool、封装流程，再让 Agent 去选择调用哪一个。

这个思路在传统机器人阶段没有问题，因为那时候重点是把几个高频场景做好。

但现在问题已经变了。

现在用户面对的是 Agent，不是菜单。

用户不会先想“我要进入会议场景”或者“我要进入文档场景”，用户只会直接说目标，比如：

- 帮我把昨天几个会的内容整理一下，生成纪要，拆出待办，再同步到群里。
- 帮我查一下这个同事最近有没有空，把下周可约时间列出来，顺手建个会。
- 帮我把这个表格里的异常项整理成文档，然后艾特相关同学处理。

这种请求天然就是跨模块的。

消息、日程、任务、文档、表格，是混在一起出现的。

如果我们还是按场景去组织系统，Agent 第一步不是执行任务，而是先猜“这个需求更像哪个场景”。

一旦用户的表达刚好跨过几个场景边界，系统就会开始绕。

这就是我觉得 `lark-cli` 这套方式更适合后续建设的根本原因。

它不是先问用户属于哪个场景，而是先把飞书能力收敛成统一的原子动作。

用户提目标，Agent 去拆步骤。

消息就是消息，建文档就是建文档，查日程就是查日程，写任务就是写任务。

这些动作本身有明确输入、有明确输出、有明确身份要求，也有明确权限要求。

这套方式更接近现在智能体真正需要的能力组织方式。

说得直接一点，用户提目标，Agent 负责拆动作和执行，不应该再让 Agent 去猜我们内部到底预设了多少个场景入口。

## 为什么说它对 Agent 更友好

从 Agent 的角度看，这个差异更明显。

在现在这套方案里，Agent 其实承担了很多本来不该由它承担的成本。

它不只是在理解用户任务，它还要理解系统结构。

它要知道系统里有哪些工具，它们分别服务于什么场景，它们的隐含前提是什么，同一个动作为什么在不同场景里可能被封装成不同入口。

遇到失败的时候，它还要自己判断这到底是权限问题、身份问题、参数问题，还是选错场景了。

这件事看起来只是“封装方式不同”，但实际影响很大。

因为 Agent 的推理预算是有限的。

如果它的大量精力都花在理解项目内部结构、理解场景边界、理解权限约束上，那它真正留给任务理解和执行编排的空间就会被压缩掉。

`lark-cli` 的优势，不在于它是命令行工具，而在于它把这些隐含成本往系统层前置了。

- 能力按业务域和原子动作组织
- 输入输出相对稳定
- 身份是显式的
- scope 是显式的
- 错误是结构化的

这意味着 Agent 不需要再去猜很多事情。

它面对的不是一堆项目内部 service，而是一组清楚的动作。

它不用先问“系统里有没有会议后跟进这个场景”，它只需要决定下一步是先查会议、再写文档，还是先建任务。

这两种工作方式对模型的压力完全不是一个量级。

所以如果问我，为什么说这套方式对 Agent 更友好，我会给一个很直接的回答：

- 现在的方案，更像是让 Agent 去适应系统
- `lark-cli` 这套方式，更像是先把系统整理成 Agent 容易使用的形态

## 权限体系为什么是核心问题

除了能力组织方式，权限体系也是一个绕不过去的问题。

这一点我觉得要单独讲，因为它不是附属问题，而是现有方案里一个长期复杂度来源。

现在这套模型里，并没有形成完整的、面向多用户的统一授权体系。

也就是说，下面这些问题，并没有统一收敛到一层去处理：

- 某个动作到底是代表应用执行，还是代表当前用户执行
- 某个用户到底有没有授权某项能力
- 多用户之间的数据和调用边界怎么隔离

结果是什么。

结果就是权限问题被拆散了。

- 一部分落在场景设计里
- 一部分落在工具封装里
- 一部分落在 Agent 推理里

每做一个新场景，都要重新想一次权限边界。

每接一类新能力，都要重新想一次这个动作到底应该走应用身份还是用户身份。

每接一个 Agent，都要重新考虑一遍这个工具放出去会不会越权。

这件事短期还能靠人盯着，长期一定会变成负担。

因为你会发现，系统越大，真正难的不是能力本身，而是“谁能在什么条件下调用什么能力”。

`lark-cli` 在这里提供的是一个很明确的结构。

- `bot` 和 `user` 身份是显式的
- `user` 调用背后对应的是明确的用户授权，而不是模糊的上下文假设
- 缺哪些 scope，可以预检查，也可以结构化返回
- 如果需要补授权，可以明确告诉用户补什么，补完之后继续执行

这点非常重要。

因为一旦权限模型是统一的，权限隔离就不再主要依赖场景设计时的小心处理。

它可以落到统一鉴权层去解决。

这样 Agent 就不用一边想任务，一边替系统兜权限风险。

## 这件事最终带来的业务价值

从老板更关心的几个结果看，我觉得这件事的价值主要体现在四个方面。

### 1. 迭代速度

如果飞书能力已经被整理成稳定的原子动作，那很多新需求本质上只是能力组合变化，不一定需要再新做一个 service，或者再新开一个场景。

这会直接影响后续需求响应速度。

尤其是在需求分散、变化快、经常临时冒出新组合的时候，原子能力体系明显更占优。

### 2. 灵活度

用户真实提出的问题，天然是跨模块的。

如果系统还是按场景分块，任何跨块需求都会先遇到边界问题。

如果系统底层已经是统一原子能力，消息、文档、任务、日程、表格这些能力可以直接组合，很多需求就不需要预先设计场景。

### 3. 用户满意度

用户其实不关心你底层是 service 还是 CLI。

用户关心的是三件事：

- 我说一个目标，系统能不能接住
- 缺权限的时候，系统能不能说清楚怎么补
- 补完以后，系统能不能接着做

谁能把这三件事做好，谁的体验就更像真正可用的 Agent。

### 4. 长期成本

如果继续按项目内场景堆，后面每做一个新机器人、新工作流、新 Agent，都要重复接一轮飞书能力。

如果飞书能力先沉到统一层，上层业务系统只管编排和产品逻辑，多套系统之间就能复用同一套底座。

这对后面人力投入和维护成本的影响，比眼前多写几个接口要大得多。

## 证据基础

我再说一下证据基础。

这不是一个纯判断题，我这次是做了实测的。

### 1. `lark-cli` 真实能力测试

我已经按 `bot` 和 `user` 两种身份，把 IM、Docs、Drive、Base、Sheets、Calendar、Task、Wiki、VC 等模块实际跑过。

每个模块都留了输入输出、命令、结果和错误信息。

所以至少在标准飞书环境里，它已经证明自己不是概念，而是一套可以演示、可以验证、可以复现的能力体系。

### 2. 私有化适配进度

我在 `/Users/wangqizhao/Developer/iflytek/cli` 里，已经补了这些能力：

- 私有化端点覆盖
- 私有化登录链路
- Authorization Code Flow 适配
- scope 兼容处理
- 一轮真实命令回归

这说明现在不是从零开始讨论，而是已经处在“方向可行，值得继续收敛”的阶段。

## 风险也要提前讲清楚

这个方向我建议推进，但现实风险主要不在代码，而在环境。

自部署后的飞书能力和标准飞书并不完全一致，这一点必须说透。

最后能不能达到标准飞书 demo 的完整程度，不完全由我们的代码决定，更多还是受环境能力约束。

现在最明显的两个风险：

- MCP 支持缺失或者不完整。这会直接影响一部分文档类、知识类链路的展示效果。
- `user` 身份能力受限，或者权限模型和标准飞书不一致。这会导致命令层设计没有问题，但真实链路还是可能断。

所以我建议汇报时不要把这件事包装成“做完迁移就一定能达到标准飞书的全部演示效果”。

更准确的表达应该是：能力层迁移可以解决我们自己的组织方式、Agent 接入方式和权限治理问题，但私有化环境能力不一致，仍然会限制最终展示上限。

## 为什么建议完整迁移，而不是长期双轨并行

如果我们现在只是说，保留原来的场景化体系，再在底下慢慢补原子能力，看上去比较稳，但实际上会有两个问题。

### 1. 认知模型冲突

上层还是按场景理解系统，底层却要求按原子动作组织能力。

这样做：

- Agent 看到的是混合工具面
- 研发维护的是混合封装
- 权限处理也会变成双轨

时间一长，只会越来越乱。

### 2. 重复建设不可避免

只要场景化体系继续是主路径，后面新需求一来，团队天然还是会优先去补场景。

原子能力体系就会一直停留在“有一些，但不是主干”的状态。

这样最后既没有彻底保住原有路径的简单性，也没有真正获得原子化体系的收益。

所以我的建议不是激进，而是尽早统一方向。

`spark` 作为业务系统可以继续保留，但飞书能力这一层应该完整迁到统一能力层上。

之后 `spark` 做它该做的事：

- 渠道接入
- Agent 运行时
- 业务编排
- 产品逻辑

底层飞书动作统一交给能力层处理。

这样分层才清楚，后面的演进路径也才清楚。

如果要落地，我建议节奏上可以分阶段，但目标上不要模糊。

也就是实施可以分步，方向不要双轨。

- 先做盘点
- 再做映射
- 再做适配
- 最后切主路径

但最终目标应该明确，就是把飞书能力从项目内场景封装，迁到统一原子能力体系上。

## 结尾

最后我用一句话收一下。

这次我看到的，不是“一个 CLI 工具能不能用”，而是两种完全不同的能力组织方式。

- 前一种方式，是继续围绕场景封装系统，让 Agent 去适应系统。
- 后一种方式，是先把能力整理成统一原子动作，让 Agent 真正围绕用户目标去执行。

从现在这个阶段看，我认为后者更适合继续投入，而且应该明确按完整迁移来推进。

否则场景化和原子化长期并存，后面付出的整合成本只会更大。
