# 汇报 QA 文档

日期：2026-03-29

这份文档按老板视角整理，问题尽量贴近现场会被追问的方向。  
回答口径尽量直接，不绕术语。

## 1. 现在的代码里不是已经有 skills 了吗？为什么不能做到同样的原子化能力？

不能简单画等号。  
现在的 skill 更多是在现有系统之上做调用编排和能力包装，但底层飞书能力本身并没有真正收敛成统一原子层。

差别在于：

- skill 是上层使用方式。
- 原子能力层是底层能力组织方式。

如果底层还是按场景、按 service、按项目内部封装组织，那 skill 再多，也只是把这些不统一的能力重新包装一遍。  
它可以缓解使用问题，但解决不了底层能力模型不统一的问题。

更直接一点说：

- 现在的 skill，更像是在一堆现有能力上做“组装”。
- `xfchat_cli` 的原子化能力，是先把底层动作本身做统一，再让 skill 去编排。

这两者的复用深度不一样。  
前者复用的是“现有封装”，后者复用的是“统一动作定义”。

## 2. 如果 skill 也能继续往细拆，为什么还要迁移？直接在现有体系里改不行吗？

理论上可以继续拆，但问题不在“能不能拆”，而在“拆完之后底层是不是统一的”。

如果继续在现有体系里拆 skill，通常会碰到几个问题：

- skill 依赖的底层 service 仍然分散在不同项目模块里。
- 相似动作可能在不同场景下有不同入口和不同参数风格。
- 权限、身份、错误处理仍然没有统一收敛。
- 新 Agent 接入时，还是要先理解项目内部封装，而不是只理解统一能力接口。

也就是说，只在 skill 层做细化，往往会得到“更细的包装层”，但不一定得到“统一的能力层”。

## 3. 现在这套方案已经能跑，为什么一定要完整迁移？渐进式共存不行吗？

短期共存可以作为实施手段，但不适合作为长期目标。

原因是场景化体系和原子化体系的出发点不同：

- 场景化体系强调“围绕预设流程封装能力”。
- 原子化体系强调“围绕通用动作组织能力”。

这两套体系长期并存，会出现几个后果：

- 新需求来了，团队会在两套路径之间摇摆。
- 同一类能力会出现双份封装。
- 权限和身份处理会形成双轨。
- Agent 面对的工具面会越来越混乱。

所以实施上可以分阶段，但目标上应该明确，就是最终收敛成一套主路径。

## 4. 你说 `xfchat_cli` 更 AI-native，这会不会只是换了个说法？

不是概念替换，差别是很具体的。

现在的方式，核心还是先设计场景，再让 Agent 去匹配场景。  
`xfchat_cli` 的方式，是用户直接提目标，Agent 再去拆动作和执行动作。

两者的核心区别不是名字，而是执行路径不同：

- 现在更像“目标 -> 找场景 -> 调工具”。
- `xfchat_cli` 更像“目标 -> 拆动作 -> 连续执行”。

如果用户需求经常跨模块，这两种模式的体验差异会很明显。

## 5. 为什么你认为现在的方式对 Agent 不够友好？

因为 Agent 现在要承担很多本来不该由它承担的理解成本。

它不只是在理解用户任务，还要理解：

- 系统里有哪些场景入口
- 哪些工具属于哪个场景
- 每个工具背后有什么隐含前提
- 失败时到底是权限问题还是场景不匹配

这些成本本质上是在消耗 Agent 的推理预算。  
而原子能力体系的价值，就是把这些信息尽量显式化、结构化，让 Agent 处理更少的系统细节。

## 6. 现在的 Agent 不也能调工具吗？为什么还说推理压力会下降？

能调工具，不等于调得轻松。

推理压力下降，主要来自三点：

- 工具面更统一，不需要先在多套场景封装之间做判断。
- 输入输出更稳定，不需要反复适配不同 service 的风格。
- 身份、scope、错误更明确，失败后更容易决定下一步。

所以不是说 Agent 以前不能做，而是说以前为了做成一件事，要在系统理解上多绕很多弯。

## 7. 为什么你反复强调权限体系？这不是实现细节吗？

不是实现细节，这是系统能不能长期稳定演进的核心问题之一。

现在最大的隐患不是“有没有这个功能”，而是“这个功能到底能代表谁执行”。  
如果没有统一的用户授权体系，权限隔离就只能分散在场景设计和工具封装里临时处理。

这会带来几个问题：

- 多用户边界不清楚
- `bot` 和 `user` 调用容易混淆
- 新增场景时要反复补权限判断
- Agent 容易被迫承担越权判断压力

`xfchat_cli` 的价值在于，它把这件事明确成了统一模型，而不是让每个场景各自处理。

## 8. 现在是不是没有用户授权体系？

不是完全没有，而是没有形成统一、清晰、可复用的用户授权模型。

现状更接近：

- 某些地方有用户上下文
- 某些地方有权限判断
- 某些能力可以区分身份

但这些东西没有被统一成一层标准机制。  
所以每做一个新能力，还是要重新考虑“按谁执行、怎么授权、如何隔离”。

这和统一的 `user` 授权模型不是一回事。

## 9. 如果 `xfchat_cli` 也有 `bot` / `user` 区分，是不是复杂度也一样高？

不是一个层面的复杂度。

复杂度无法消失，但可以集中治理。  
`xfchat_cli` 里，`bot` / `user` 身份、scope、权限错误、补授权逻辑，至少是统一表达的。  
这样复杂度是被收敛到能力层里的。

现在的问题是，复杂度分散在多个场景、多个 service、多个 skill 里。  
分散的复杂度比集中的复杂度更难治理。

## 10. 既然 `spark` 现在是业务系统，为什么不能继续让它把飞书能力也一起管着？

短期可以，长期不划算。

因为 `spark` 的价值本来就在业务编排、运行时和产品逻辑。  
如果飞书能力也继续深埋在里面，后面每扩一个机器人、一个 Agent、一条新链路，都要重复接一次飞书能力。

这样做的结果不是系统更完整，而是业务层越来越重，复用越来越差。

更合理的分工应该是：

- `spark` 管业务和运行时
- 统一能力层管飞书原子动作

这样边界会更清楚。

## 11. 迁移之后，真的能提升响应速度吗？会不会只是架构上更好看？

不会只停留在架构层面。

响应速度提升不一定表现为单次 API 更快，而是表现为整条任务链路更短、更少绕路。

主要体现在：

- 不需要先匹配场景，再找场景内工具
- 不需要在多套封装之间反复切换
- 错误类型更清晰，失败后恢复更快
- 跨模块动作可以直接组合，不用额外桥接

所以它提升的是任务完成效率，而不是单个 HTTP 请求耗时。

## 12. 迁移之后，真的能降低模型推理压力吗？怎么证明？

这是一个工程判断，不是数学定理。  
但从调用路径可以清楚看出原因。

如果 Agent 需要先判断：

- 用户属于哪个场景
- 当前该选哪个业务 tool
- 失败后该回退到哪个场景

那模型消耗的上下文和推理步骤一定更多。

如果 Agent 面对的是统一原子动作，它更多是在做步骤编排，而不是系统结构判断。  
这类任务天然更短、更直接，也更稳定。

## 13. 为什么说跨模块原子能力很重要？我们现在也能做跨模块流程。

现在也能做，但大多是“为某个流程专门写一条跨模块链路”。  
这和“天然支持跨模块原子组合”不是一回事。

差别在于：

- 现在是先定义流程，再连能力。
- 原子能力体系是先有统一动作，再让流程自由组合。

前者适合少量稳定流程。  
后者更适合需求不断变化、组合不断变化的环境。

## 14. 如果最终还是要写 workflow，那底层是不是原子能力其实没区别？

有区别，而且区别很大。

workflow 解决的是“怎么组合”。  
原子能力解决的是“拿什么组合”。

如果底层能力本身不统一，workflow 只是把不统一的东西重新串了一遍。  
如果底层能力已经统一，workflow 才能真正做到低成本复用。

所以 workflow 不能替代统一能力层。

## 15. 这件事为什么不是优化一下现有工具面就够了？

因为问题不只在工具展示层，而在能力组织层。

如果只是把工具名改得更统一、入口藏得更深，但底层还是按场景和项目内部 service 组织，那后面的问题还会继续出现：

- 权限判断重复做
- 相似动作重复封装
- Agent 仍然要理解系统内部结构

所以这不是工具面整理问题，而是底层能力模型问题。

## 16. 你为什么建议“完整迁移”，而不是“先试点看看”？

试点当然可以做，但试点应该是实施节奏，不应该成为长期架构状态。

我建议完整迁移，是因为：

- 场景化和原子化长期并存，本身就是冲突状态
- 只要场景化还是主路径，新需求就会继续优先堆到场景里
- 原子能力体系如果不是主干，永远只能停留在辅助层

所以可以先试点，但目标要明确，就是最终切主路径。

## 17. 如果要完整迁移，现有 skill / tool / service 会不会基本都要重写？

不会全部重写，但会重分层。

大致可以分三类：

- 飞书原子能力，迁到统一能力层
- 业务组合能力，保留在 `spark` 或业务层
- 非飞书能力，继续独立保留

所以不是把所有东西推翻重写，而是把职责重新摆正。

## 18. 最大的不确定性是什么？

最大的不确定性不是代码实现，而是私有化环境能力差异。

当前最现实的两个点：

- MCP 支持缺失或不完整
- `user` 身份能力受限，或者权限模型与标准飞书不一致

这意味着即使架构方向正确，私有化环境下的演示上限也可能被环境本身限制住。  
这一点要提前对齐预期。

## 19. 如果私有化环境能力不完整，那做这件事是不是意义就下降了？

不会下降到没有意义，只是收益表现形式会变。

即使私有化环境不能完全达到标准飞书的演示效果，这次迁移依然可以解决几件我们自己的问题：

- 飞书能力组织方式不统一
- Agent 接入复杂度高
- 权限治理分散
- 重复封装成本高

也就是说，环境会限制展示上限，但不会否定能力层收敛本身的价值。

## 20. 如果一句话回答，为什么现在该投资源做这件事？

因为再往前做，问题已经不是“飞书功能够不够多”，而是“飞书能力是不是还要继续分散在业务系统里”。  
如果现在不统一，后面机器人越多、Agent 越多、私有化环境越多，重复建设和整合成本只会越来越高。

## 21. 如果一句话回答，完整迁移的目标到底是什么？

把飞书能力从“项目内场景封装”迁到“统一原子能力层”，让上层 Agent 和业务系统都围绕同一套动作体系工作。

## 22. 如果一句话回答，`spark` 在迁移后还剩什么价值？

`spark` 继续承接渠道接入、运行时、业务编排和产品逻辑，但不再承担飞书能力底座的职责。

## 23. 如果老板追问：你现在最担心什么？

我最担心的不是方向判断错，而是方向判断对了以后，执行上又变成长期双轨。  
因为一旦双轨长期存在，团队就会持续在场景化和原子化之间来回摇摆，最后两套体系都维护得很重。

## 24. 如果老板追问：你现在最有把握的是什么？

我最有把握的是，这不是一个纯概念判断。  
`xfchat_cli` 的真实能力我已经测过，私有化适配我也已经做了一轮，所以现在讨论的不是“想不想做”，而是“要不要正式按这个方向收敛”。
